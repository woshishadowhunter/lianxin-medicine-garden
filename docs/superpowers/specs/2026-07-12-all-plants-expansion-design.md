# 全植物种植记录与积分系统设计

## 目标

将“连心药园”从面向本草种植的专用小程序，扩展为适用于花卉、绿植、蔬菜、果树、本草及其他植物的家庭种植记录与积分平台。家庭既可以从管理员维护的预设植物库中选择，也可以创建只属于本家庭的自定义植物；旧本草任务、养护记录、照片和积分必须继续可用。

## 设计原则

- 采用兼容式扩展，不要求部署方一次性迁移旧数据库。
- 用户可见界面统一使用“植物”“植物园”“成长记录”等通用语义。
- 植物照片和成长观察是主内容，积分是持续记录的反馈，不主导整个界面。
- 自定义植物必须通过云函数创建，客户端不能为其他家庭写入任务。
- 审核通过后发放积分的既有幂等与冲正逻辑保持不变。
- 不引入第三方运行时依赖，不使用 SVG；界面资源使用 PNG/WebP 位图。

## 视觉方向

采用“自然观察手账”风格：低饱和纸张底色、深橄榄绿文本与主操作色、照片式植物封面、细边框和紧凑观察标签。减少大面积纯绿色和重复卡片，不使用卡通 SVG 插画。首页以近期真实养护照片为第一视觉信号，照片下方展示日期、植物名称和观察摘要。

## 信息架构

### 首页

- 品牌从“连心药园”改为“连心植物园”。
- 顶部显示家庭编号、社区、积分余额和连续记录概况。
- 主体优先展示最近一条已确认记录的照片、植物名称、日期和描述。
- 展示待养护提醒、最近成长记录和植物分类概况。
- 主操作是“拍照记录成长”，次级入口为“我的植物”“积分账户”。

### 植物园

- 原“药园”标签改为“植物园”。
- 顶部显示植物总数、生长中、需关注和已收获数量。
- 支持全部、花卉、绿植、蔬菜、果树、本草、其他七种分类筛选。
- 每张植物卡展示封面、名称、分类、种植天数、最近养护、记录次数和状态。
- 提供“添加植物”入口，进入预设选择与自定义创建流程。

### 添加植物

- 默认展示预设植物库，可搜索并按类别筛选。
- 选择预设植物后填写种植日期和可选昵称，确认后创建家庭种植任务。
- 找不到目标时切换到自定义表单，填写植物名称、类别、种植日期、预计成长周期和可选封面照片。
- 同一家庭允许种植多个同名植物，任务以独立 ID 区分。
- 名称长度为 1 至 30 个字符；成长周期为 1 至 3650 天；种植日期不能晚于当天。

### 养护记录

- 第一步从家庭全部生长中植物选择目标，不再使用“选择药材”文案。
- 养护类型为浇水、修剪、施肥、除草、除虫、生长观察和其他。
- 每条记录仍要求至少一张照片，最多九张。
- 记录写入通用植物字段，并同时写入旧字段别名，保证旧管理端和历史查询兼容。

### 植物详情与成长档案

- 原药材详情和药材档案统一改为植物详情与成长档案。
- 展示封面、类别、来源、种植日期、生长天数、预计周期、状态、照片时间线和养护统计。
- 对没有明确收获周期的观赏植物，显示“已陪伴 N 天”，不显示强制收获倒计时。
- 对蔬菜、果树、本草等配置了成长周期的植物，继续显示预计周期进度。

### 积分与管理端

- 每条带照片的养护记录经管理员确认后发放 10 分。
- 管理员奖励规则调整为“成长状态优秀”“关键生长节点”“完整成长档案”“社区绿色示范”。
- 积分流水、审核列表、家庭详情、统计图表和导出文件使用植物名称与类别。
- 旧记录缺少通用字段时，从 `herb_name`、`herb_code` 回退读取，并归类为“本草”。

## 数据设计

### plants 集合

预设植物库使用新集合 `plants`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| code | string | 稳定植物编码，唯一 |
| name | string | 标准名称 |
| category | string | flower/foliage/vegetable/fruit/herb/other |
| growth_days | number | 建议成长周期；0 表示不显示收获倒计时 |
| icon_name | string | 位图图标语义名，默认 plant |
| description | string | 简短介绍 |
| is_active | boolean | 是否允许新建任务 |
| created_at | Date | 创建时间 |

初始化数据覆盖多个类别，并保留原有 11 种本草。旧 `herbs` 集合不删除，作为已有部署的兼容数据源。

### planting_tasks 扩展

新增字段：

- `plant_code`: 预设植物编码；自定义植物使用 `custom_<hash>`。
- `plant_name`: 用户看到的植物名称。
- `plant_category`: 六类植物枚举之一。
- `growth_days`: 创建任务时固化的预计周期。
- `source`: `preset` 或 `custom`。
- `cover_image`: 可选云存储 fileID。
- `owner_openid`: 创建自定义任务的用户 openid。

为兼容旧代码，新任务同时写入 `herb_code`、`herb_name` 和 `herb_icon_name`。读取任务时统一经过标准化函数：优先使用 `plant_*`，缺失时回退到 `herb_*`，旧任务默认类别为 `herb`、来源为 `legacy`。

### care_records 扩展

新增 `plant_code`、`plant_name`、`plant_category`。提交记录时从服务端任务文档获取这些值，不能信任客户端传入的植物名称和家庭编号。旧 `herb_*` 字段继续同步写入。

## 云端接口

新增 `plantManager` 云函数：

- `listCatalog`: 返回启用的预设植物库，不要求家庭身份。
- `createPresetTask`: 校验当前 openid 对家庭的绑定权限，按植物编码读取预设配置并创建任务。
- `createCustomTask`: 校验家庭权限、名称、类别、日期、周期和封面 fileID 后创建任务。

`submitRecord` 调整为根据 `taskId` 读取任务并验证任务属于当前家庭，随后写入标准化植物字段。积分、审核和导出继续使用既有接口，只调整通用字段读取和显示文案。

## 权限与错误处理

- 家庭成员权限沿用 `family_members`，并兼容旧版 `families.openid` 绑定。
- 管理员可以查看所有植物任务，但家庭只能创建和查看本家庭任务。
- 自定义名称去除首尾空白并限制长度；类别必须属于固定枚举。
- 预设植物不存在或已停用时拒绝创建。
- 云函数失败时页面保留表单内容并展示可理解的错误，不产生半完成任务。
- 同一次创建请求携带 `requestId`，云端使用确定性文档 ID 防止网络重试造成重复任务。
- 离线养护队列继续工作；同步时由服务端重新校验任务归属和当前状态。

## 兼容策略

- 不删除或重命名现有集合与字段。
- 所有读取逻辑通过标准化工具处理新旧任务和记录。
- 初始化脚本为新部署创建 `plants`，并继续写入兼容字段。
- 旧部署可单独运行迁移脚本补充通用字段；未运行迁移也不影响功能。
- 原有页面路径暂时保留，避免收藏路径、页面跳转和分包配置失效；文件内部变量逐步使用 plant 语义。

## 测试策略

- 为任务标准化、类别校验、日期校验和幂等任务 ID 编写 Node 单元测试。
- 为 `plantManager` 的预设创建、自定义创建、越权访问和重复请求编写云函数测试。
- 为 `submitRecord` 增加旧任务与新任务两类字段写入测试。
- 对积分规则通用化、旧本草流水回退描述和新植物描述编写回归测试。
- 对所有修改的 JavaScript 文件运行 `node --check`。
- 静态扫描用户可见文件，确保不再出现“药园”“选择药材”“药材档案”等专用文案；数据库兼容代码与历史文档可以保留必要的 `herb_*` 标识。
- 在微信开发者工具中验证首页、植物分类、添加预设植物、添加自定义植物、提交养护、审核入账和积分流水完整流程。

## 完成标准

- 家庭可以从预设库添加植物，也可以创建任意自定义植物。
- 花卉、绿植、蔬菜、果树、本草和其他植物均可提交带照片养护记录。
- 审核通过后积分正确入账，退回修正时正确冲正。
- 旧本草任务、记录、照片和积分可以正常查看。
- 首页、植物园、详情、记录、积分和管理端均使用通用植物语义。
- 视觉采用自然观察手账方向，主要资源为 PNG/WebP，不新增 SVG。
- 自动化检查通过，代码提交到功能分支并推送到 GitHub，创建草稿合并请求。
